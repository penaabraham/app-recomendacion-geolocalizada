	\section{Justificación}
	
	El crecimiento exponencial de la información digital y la masificación de dispositivos móviles con capacidades de geolocalización han transformado la manera en que los usuarios interactúan con productos y servicios. En este contexto, los sistemas de recomendación han adquirido un papel fundamental al facilitar la toma de decisiones en entornos con alta sobrecarga informativa (\cite{revision_sr_turismo}).
	
	Si bien existen múltiples propuestas algorítmicas en la literatura, muchas de ellas se mantienen en un nivel conceptual o experimental, sin integrarse completamente en sistemas funcionales que operen con datos en tiempo real y condiciones técnicas reales. En particular, la incorporación de geolocalización dinámica dentro de sistemas híbridos de recomendación representa un desafío de integración arquitectónica, procesamiento y validación técnica.
	
	El algoritmo híbrido propuesto por Martínez Simón (\cite{dael_recomendacion_hibrida}) constituye una base sólida desde el punto de vista metodológico; sin embargo, su implementación en un entorno con geolocalización activa y arquitectura cliente-servidor requiere procesos de adaptación, integración y validación que no han sido documentados extensamente.
	
	En consecuencia, el presente proyecto se justifica como una aplicación práctica y validación técnica de un algoritmo híbrido existente, integrándolo dentro de un sistema funcional que incorpora información geográfica en tiempo real, permitiendo evaluar su comportamiento en un entorno operativo.
	
	\section{Objetivo}
	
	\subsection{Objetivo general}
	Desarrollar e integrar un sistema de recomendación híbrido basado en el algoritmo propuesto por Martínez Simón, incorporando geolocalización en tiempo real dentro de una arquitectura cliente-servidor, con el fin de validar técnicamente su funcionamiento en un entorno operativo.
	
	\subsection{Objetivos específicos}
	
	\begin{itemize}
		\item Analizar el algoritmo híbrido propuesto en la literatura para identificar sus componentes funcionales y requerimientos de implementación.
		
		\item Diseñar una arquitectura de software que permita la integración del algoritmo dentro de un sistema distribuido con backend y aplicación móvil.
		
		\item Incorporar el procesamiento de coordenadas geográficas en tiempo real como variable contextual dentro del proceso de recomendación.
		
		\item Implementar los módulos necesarios para la obtención, procesamiento y almacenamiento de datos de usuarios y productos.
		
		\item Validar el funcionamiento del sistema mediante pruebas técnicas que permitan evaluar la coherencia y estabilidad de las recomendaciones generadas.
	\end{itemize}

	\section{Marco Teórico}
	
	\subsection{Sistemas de recomendación}
	Los sistemas de recomendación constituyen un área de investigación y desarrollo ampliamente estudiada dentro de la Inteligencia Artificial y los Sistemas de Información, cuyo objetivo principal es asistir a los usuarios en la toma de decisiones mediante la sugerencia personalizada de productos, servicios o contenidos. Estos sistemas buscan reducir la sobrecarga de información y mejorar la experiencia del usuario a partir del análisis de datos históricos y contextuales (\cite{revision_sr_turismo}).
	
	De manera general, un sistema de recomendación se apoya en la identificación de patrones de comportamiento del usuario, los cuales pueden derivarse de interacciones explícitas, como calificaciones, o implícitas, como registros de navegación, compras o visitas a ubicaciones específicas (\cite{rodas_sr_colaborativo}). Este enfoque ha demostrado ser efectivo en múltiples dominios, incluyendo comercio electrónico, educación y turismo.
	
	\subsection{Filtrado colaborativo}
	El filtrado colaborativo es una de las técnicas más representativas dentro de los sistemas de recomendación. Su principio fundamental consiste en asumir que usuarios con comportamientos similares en el pasado tenderán a mostrar preferencias semejantes en el futuro. Bajo este enfoque, las recomendaciones se generan a partir del análisis de interacciones entre usuarios y elementos, sin necesidad de describir explícitamente las características de los productos (\cite{rodas_sr_colaborativo}).
	
	Diversos estudios han evidenciado que el filtrado colaborativo basado en usuarios ofrece resultados adecuados cuando existe suficiente información histórica, especialmente en contextos donde las preferencias pueden inferirse a partir de patrones de uso compartidos (\cite{dael_recomendacion_hibrida}). No obstante, este enfoque presenta limitaciones conocidas, como el problema de arranque en frío y la escasez de datos.
	
	\subsection{Sistemas de recomendación híbridos}
	Con el objetivo de mitigar las limitaciones inherentes a los enfoques individuales, surgen los sistemas de recomendación híbridos, los cuales combinan múltiples técnicas dentro de un mismo modelo. Esta integración permite aprovechar las fortalezas de cada método y mejorar la calidad global de las recomendaciones (\cite{revision_sr_turismo}).
	
	En particular, los sistemas híbridos que integran filtrado colaborativo con técnicas adicionales, como análisis de contenido, clustering o información contextual, han mostrado un desempeño superior frente a enfoques tradicionales. El algoritmo propuesto por Martínez Simón constituye un ejemplo de este tipo de aproximaciones, al integrar diferentes fuentes de información para generar recomendaciones más precisas (\cite{dael_recomendacion_hibrida}).
	
	\subsection{Agrupamiento y técnicas de clustering}
	El clustering o agrupamiento es una técnica de aprendizaje no supervisado que permite organizar datos en grupos homogéneos a partir de criterios de similitud. Dentro del contexto de los sistemas de recomendación, el clustering facilita la segmentación de usuarios o elementos, reduciendo la complejidad computacional y mejorando la eficiencia del proceso de recomendación (\cite{clusterdoc_agrupamiento}).
	
	Herramientas como K-Means han sido ampliamente utilizadas para identificar patrones latentes en conjuntos de datos y apoyar procesos de recuperación y recomendación de información. La utilización de técnicas de agrupamiento resulta especialmente útil cuando se busca integrar información adicional, como el contexto geográfico, dentro del modelo de recomendación (\cite{clusterdoc_agrupamiento}).
	
	\subsection{Recomendación basada en geolocalización}
	La incorporación de información geográfica en los sistemas de recomendación ha cobrado relevancia con el auge de los dispositivos móviles y los servicios basados en localización. En este tipo de sistemas, la ubicación del usuario se convierte en un factor determinante para mejorar la pertinencia de las recomendaciones, especialmente en escenarios donde la proximidad física influye directamente en la decisión de consumo (\cite{revision_sr_turismo}).
	
	Diversos trabajos aplicados han demostrado que la utilización de la localización permite ofrecer recomendaciones más contextualizadas y útiles, al reducir el espacio de búsqueda y priorizar elementos cercanos al usuario. Este enfoque ha sido utilizado en entornos comerciales y de servicios, donde la dimensión espacial forma parte inherente del proceso de recomendación (\cite{supermercado_recomendacion_localizacion}).
	
	\subsection{Aplicaciones prácticas de los sistemas de recomendación}
	
	La aplicación de sistemas de recomendación en entornos reales ha permitido validar su utilidad más allá del ámbito teórico. Proyectos como el desarrollado por Calderón Pacheco y Vega Asto evidencian cómo la integración de patrones de comportamiento y localización puede mejorar la experiencia del usuario en contextos comerciales físicos, como supermercados (\cite{supermercado_recomendacion_localizacion}).
	
	Estos trabajos refuerzan la idea de que el valor de un sistema de recomendación no radica únicamente en la novedad del algoritmo empleado, sino en su correcta adaptación, integración y validación dentro de un sistema funcional. Bajo esta perspectiva, el presente proyecto se enmarca como una aplicación práctica de un algoritmo híbrido existente, adaptado a un entorno móvil con geolocalización en tiempo real.

	\section{Metodología}
	
	El presente proyecto se enmarca dentro de la investigación aplicada con enfoque
	experimental, dado que su propósito central no es generar nuevo conocimiento
	teórico, sino adaptar, integrar y validar técnicamente un algoritmo de
	recomendación híbrido existente en un entorno operativo real. El enfoque
	experimental se justifica en la medida en que se construyó un sistema funcional
	sobre el cual se realizaron pruebas controladas para observar el comportamiento
	del algoritmo bajo distintas condiciones de entrada, particularmente en lo
	referente a la variación de coordenadas geográficas.
	
	\subsection{Arquitectura del sistema}
	
	El sistema se diseñó bajo una arquitectura cliente-servidor distribuida en tres
	capas claramente delimitadas: el frontend móvil, el backend de procesamiento y
	el módulo del algoritmo de recomendación.
	
	El \textbf{frontend} fue desarrollado como una aplicación móvil multiplataforma
	utilizando Flutter. Su responsabilidad principal es obtener las coordenadas
	geográficas del dispositivo mediante el servicio GPS, enviarlas al backend a
	través de una solicitud HTTP, y presentar los resultados al usuario en forma de
	lista ordenada con información de distancia, calificación y razones de
	recomendación. La interfaz incorpora además una barra de búsqueda con filtrado
	en tiempo real y paginación progresiva de resultados.
	
	El \textbf{backend} fue implementado con FastAPI, un framework de Python de alto
	rendimiento para el desarrollo de APIs REST. Expone un único endpoint
	\texttt{GET /recomendar/\{user\_id\}} que recibe los parámetros de latitud,
	longitud y un límite opcional de resultados. Al arrancar el servidor, el backend
	carga en memoria todos los modelos previamente serializados por un script de
	entrenamiento independiente (\texttt{train.py}), de modo que las solicitudes
	en tiempo real no incurren en costos de reentrenamiento. Ante cualquier
	excepción, el backend retorna un objeto JSON con la clave
	\texttt{error\_interno}, permitiendo al frontend distinguir errores del
	algoritmo de errores de red.
	
	El \textbf{módulo del algoritmo} opera como una biblioteca interna invocada por
	el backend. Concentra toda la lógica de cálculo de scores, normalización,
	combinación ponderada y generación de razones de recomendación, sin
	dependencias hacia la capa de presentación. Esta separación permite modificar
	o sustituir el algoritmo sin afectar al resto del sistema.
	
	La comunicación entre capas se realiza exclusivamente mediante JSON sobre HTTPS,
	lo que garantiza interoperabilidad e independencia tecnológica entre el cliente
	móvil y el servidor.
	
	\subsection{Descripción del algoritmo híbrido}
	
	El algoritmo implementado sigue la propuesta de Martínez Simón
	\autocite{dael_recomendacion_hibrida} y combina cuatro fuentes de información
	en un score final ponderado. A continuación se describe cada componente y su
	contribución al resultado.
	
	\subsubsection*{Score geográfico}
	
	Para cada producto $p$ con coordenadas $(\phi_p, \lambda_p)$ y un usuario
	ubicado en $(\phi_u, \lambda_u)$, la distancia física se calcula mediante la
	fórmula de Haversine:
	
	\begin{equation}
		d = 2R \arctan2\!\left(\sqrt{a},\, \sqrt{1-a}\right)
		\label{eq:haversine}
	\end{equation}
	
	\noindent donde $R = 6{,}371$ km es el radio medio terrestre y
	
	\begin{equation}
		a = \sin^2\!\left(\frac{\Delta\phi}{2}\right)
		+ \cos\phi_u \cos\phi_p \sin^2\!\left(\frac{\Delta\lambda}{2}\right)
		\label{eq:haversine_a}
	\end{equation}
	
	La distancia se normaliza de forma inversa para que los productos más cercanos
	obtengan un score mayor:
	
	\begin{equation}
		\text{geo\_score} = 1 - \frac{d - d_{\min}}{d_{\max} - d_{\min} + \varepsilon}
		\label{eq:geo_score}
	\end{equation}
	
	\subsubsection*{Score de calificación}
	
	El promedio de calificaciones por producto se obtiene a partir de las
	interacciones históricas y se normaliza en el rango $[0, 1]$:
	
	\begin{equation}
		\text{rating\_score} = \frac{\bar{r} - \bar{r}_{\min}}{\bar{r}_{\max} - \bar{r}_{\min} + \varepsilon}
		\label{eq:rating_score}
	\end{equation}
	
	\subsubsection*{Score de contenido}
	
	El score de contenido integra tres señales derivadas del análisis de los
	comentarios asociados a cada producto:
	
	\begin{itemize}
		\item \textbf{TF-IDF}: los comentarios de cada producto se agrupan y
		vectorizan mediante TF-IDF. La similitud coseno promedio entre el vector
		de un producto y el resto del catálogo sirve como indicador de relevancia
		textual.
		
		\item \textbf{LDA} (\textit{Latent Dirichlet Allocation}): se entrena un
		modelo de cinco tópicos sobre la matriz TF-IDF. La intensidad del tópico
		dominante por producto, una vez normalizada, aproxima la coherencia
		temática del contenido asociado.
		
		\item \textbf{Score de sentimiento}: un modelo de embeddings entrenado con
		Keras clasifica los comentarios como positivos (calificación $> 3$) o
		negativos. La probabilidad de sentimiento positivo, agregada por producto
		y normalizada, refleja el tono general de las opiniones de los usuarios.
	\end{itemize}
	
	El score de contenido combinado se calcula como:
	
	\begin{equation}
		\text{content\_score} = 0.5 \cdot \text{tfidf\_score}
		+ 0.2 \cdot \text{lda\_score}
		+ 0.3 \cdot \text{sentiment\_score}
		\label{eq:content_score}
	\end{equation}
	
	\subsubsection*{Score final}
	
	Los tres componentes se integran en un score final ponderado:
	
	\begin{equation}
		\text{final\_score} = 0.40 \cdot \text{geo\_score}
		+ 0.25 \cdot \text{rating\_score}
		+ 0.35 \cdot \text{content\_score}
		\label{eq:final_score}
	\end{equation}
	
	Los pesos fueron calibrados de forma empírica con el objetivo de otorgar
	prioridad a la proximidad geográfica como factor contextual determinante,
	al tiempo que se preserva una contribución significativa del análisis textual,
	reconociendo que la calidad percibida del producto no se reduce únicamente
	a su calificación numérica.
	
	Los productos se ordenan de mayor a menor \texttt{final\_score} y se
	devuelven al usuario acompañados de una etiqueta explicativa generada de
	forma automática a partir de los umbrales individuales de cada score, con el
	propósito de hacer transparente el criterio de recomendación.
	
	\subsection{Tecnologías utilizadas}
	
	El desarrollo del sistema implicó la integración de tecnologías en tres
	dominios: aplicación móvil, backend y procesamiento de datos.
	
	En el \textbf{frontend} se utilizó Flutter con el lenguaje Dart. La elección
	se fundamenta en su capacidad multiplataforma (Android e iOS desde una única
	base de código) y en la disponibilidad de paquetes como \texttt{geolocator}
	para la obtención de coordenadas GPS y \texttt{http} para la comunicación
	con el backend.
	
	En el \textbf{backend} se empleó \texttt{FastAPI 0.110.0} sobre Python 3,
	servido mediante \texttt{uvicorn 0.28.0}. Su sistema de tipado mediante
	anotaciones permite la validación automática de parámetros de entrada y la
	generación de documentación interactiva, lo que facilitó las pruebas durante
	el desarrollo. El servidor fue desplegado en la plataforma Render, accesible
	mediante HTTPS.
	
	Para el \textbf{procesamiento de datos y modelado}, se utilizaron las
	siguientes bibliotecas del ecosistema científico de Python:
	
	\begin{itemize}
		\item \texttt{pandas 2.2.3} y \texttt{NumPy 1.26.4}: manipulación de
		datos tabulares y operaciones vectoriales.
		\item \texttt{scikit-learn 1.8.0}: vectorización TF-IDF, similitud coseno,
		K-Means y LDA.
		\item \texttt{tensorflow-cpu 2.16.1} / \texttt{Keras}: construcción y
		entrenamiento del modelo de embeddings para análisis de sentimiento. Se
		utilizó la variante \texttt{cpu} dado que el entorno de despliegue no
		cuenta con GPU dedicada.
		\item \texttt{joblib 1.5.3} y \texttt{pickle}: serialización de modelos
		entrenados para su carga en tiempo de ejecución sin reentrenamiento.
	\end{itemize}
	
	\subsection{Conjunto de datos}
	
	Dado que el proyecto se desarrolla en un contexto académico con restricciones
	de privacidad, no se utilizaron datos reales de usuarios ni de productos. En
	su lugar, se generó un conjunto de datos sintéticos mediante un script de
	Python diseñado específicamente para este propósito.
	
	El conjunto de datos se compone de cinco archivos en formato JSON:
	
	\begin{itemize}
		\item \texttt{users.json}: 60 registros de usuarios, cada uno con un
		campo \texttt{user\_id} como identificador único entero.
		
		\item \texttt{locations.json}: 60 registros de coordenadas geográficas,
		vinculados a cada usuario mediante \texttt{user\_id}. Los valores de
		latitud y longitud se generaron de forma aleatoria uniforme dentro de
		los rangos $[19.2,\ 19.6]$ y $[-99.4,\ -98.9]$ respectivamente,
		correspondientes a una zona de la región centro de México.
		
		\item \texttt{products.json}: 150 productos con campos \texttt{item\_id},
		\texttt{name} y coordenadas geográficas distribuidas en la misma zona
		que los usuarios. Los nombres se construyeron combinando una marca y
		modelo de una lista de 20 categorías de productos electrónicos y del
		hogar con un sufijo numérico único.
		
		\item \texttt{ratings.json}: 900 calificaciones con campos
		\texttt{user\_id}, \texttt{item\_id}, \texttt{rating} (entero en escala
		1--5, distribución uniforme discreta) y \texttt{timestamp} (marca
		temporal Unix con desplazamiento aleatorio de hasta $10^6$ segundos
		hacia el pasado).
		
		\item \texttt{comments.json}: 250 comentarios textuales con campos
		\texttt{item\_id} y \texttt{comment}. Cada comentario se seleccionó
		aleatoriamente de una plantilla de diez frases en español con
		valoraciones tanto positivas como negativas, con el propósito de
		proporcionar variabilidad suficiente para los modelos TF-IDF, LDA y
		de sentimiento.
	\end{itemize}
	
	Se reconoce que la naturaleza sintética de los datos constituye una
	limitación en cuanto a la validación de la precisión de las recomendaciones
	en condiciones reales. En particular, la generación de comentarios a partir
	de plantillas fijas reduce la riqueza léxica que cabría esperar en un corpus
	de opiniones auténticas, lo que puede afectar la discriminación de los
	modelos de texto. No obstante, esta decisión es consistente con el objetivo
	del proyecto, que es la validación técnica del sistema y no la evaluación
	de su efectividad comercial.

	estructura interna del contenedor Backend + Algoritmo, mostrando las seis
	unidades funcionales que lo componen. El \textbf{Endpoint de recomendación}
	actúa como punto de entrada HTTP, valida los parámetros recibidos e invoca
	los motores de cálculo. El \textbf{Cargador de Modelos} deserializa los
	artefactos de ML al iniciar el servidor, evitando reentrenamientos en tiempo
	de solicitud. Los motores \textbf{Geográfico}, \textbf{Colaborativo} y
	\textbf{de Contenido} calculan de forma independiente el \textit{geo\_score},
	el \textit{rating\_score} y el \textit{content\_score} respectivamente,
	consultando tanto el almacén de datos como los modelos cargados. Finalmente,
	el \textbf{Combinador de Scores} integra los tres valores parciales en un
	score final ponderado, ordena los productos resultantes y genera las etiquetas
	explicativas que se devuelven al usuario.

		\label{fig:Codigo}
	\end{figure}
	
	El diagrama de código (Figura~\ref{fig:Codigo}) presenta la estructura del
	almacén de datos mediante un modelo entidad-relación, describiendo los cinco
	archivos JSON que conforman el conjunto de datos sintético. La entidad
	\textbf{User} se relaciona de forma uno a uno con \textbf{Location}, dado que
	cada usuario cuenta con una única coordenada geográfica registrada. Tanto
	\textbf{User} como \textbf{Product} mantienen una relación uno a muchos con
	\textbf{Rating}, reflejando que un usuario puede calificar múltiples productos
	y un producto puede recibir múltiples calificaciones. La entidad
	\textbf{Comment} se asocia exclusivamente con \textbf{Product}, ya que el
	conjunto de datos no registra la autoría de los comentarios a nivel de usuario.
	Este modelo constituye la base sobre la cual operan los motores colaborativo y
	de contenido del algoritmo de recomendación.

	% ──────────────────────────────────────────────────────
	\subsubsection{Paso 4: Procesamiento de Datos}
	
	El procesamiento de datos comprende tres técnicas complementarias que
	alimentan el componente de contenido del algoritmo híbrido: filtrado basado
	en contenido mediante TF-IDF, cálculo de similitud coseno entre productos,
	y clasificación de sentimientos mediante un modelo de aprendizaje profundo.
	Este procesamiento ocurre íntegramente en \texttt{train.py} y sus resultados
	se persisten como artefactos serializados.
	
	\paragraph{Filtrado Basado en Contenido y Cálculo de Similitud.}
	Los comentarios asociados a cada producto se agrupan en un único documento
	por \texttt{item\_id} y se vectorizan mediante TF-IDF, técnica que pondera
	la frecuencia de cada término en relación con su presencia en el conjunto
	total de documentos. Sobre la matriz resultante se calcula la similitud
	coseno entre todos los pares de productos, obteniendo una medida de
	proximidad semántica que no depende de la longitud de los textos. El modelo
	LDA complementa este análisis al identificar cinco tópicos latentes en el
	corpus, cuya distribución por producto se emplea posteriormente como señal
	adicional en el score de contenido:
	
	\begin{lstlisting}[language=Python, caption=TF-IDF + similitud coseno + LDA en train.py]
		# Agrupar comentarios por producto
		comments_grouped = comments.groupby('item_id')['comment'] \
		.apply(' '.join).reset_index()
		
		# Vectorizacion TF-IDF
		tfidf = TfidfVectorizer(stop_words='english')
		tfidf_matrix = tfidf.fit_transform(comments_grouped['comment'])
		
		# Similitud coseno producto vs. producto
		cosine_sim      = cosine_similarity(tfidf_matrix, tfidf_matrix)
		item_ids_unique = comments_grouped["item_id"].tolist()
		similarity_df   = pd.DataFrame(cosine_sim,
		index=item_ids_unique,
		columns=item_ids_unique)
		
		# Modelado de topicos con LDA
		lda        = LDA(n_components=5, random_state=42)
		lda_matrix = lda.fit_transform(tfidf_matrix)
		
		# Serializacion
		joblib.dump(tfidf,         f"{MODELS_DIR}/tfidf.pkl")
		joblib.dump(similarity_df, f"{MODELS_DIR}/similarity_df.pkl")
		joblib.dump(lda,           f"{MODELS_DIR}/lda.pkl")
		joblib.dump(lda_matrix,    f"{MODELS_DIR}/lda_matrix.pkl")
	\end{lstlisting}
	
	De forma paralela, se construye la matriz usuario-producto a partir de las
	calificaciones y se calcula la similitud coseno entre usuarios, base del
	componente de filtrado colaborativo. Adicionalmente, se calcula el promedio
	de calificaciones por producto, utilizado en el \textit{rating\_score}:
	
	\begin{lstlisting}[language=Python, caption=Filtrado colaborativo y rating promedio en train.py]
		# Matriz usuario-producto
		rating_matrix = ratings.pivot_table(
		index='user_id', columns='item_id', values='rating'
		).fillna(0)
		
		# Similitud entre usuarios
		user_sim = cosine_similarity(rating_matrix, rating_matrix)
		
		# Promedio de calificaciones por producto
		rating_avg = ratings.groupby("item_id")["rating"].mean().reset_index()
		rating_avg.rename(columns={"rating": "avg_rating"}, inplace=True)
		
		joblib.dump(rating_matrix, f"{MODELS_DIR}/rating_matrix.pkl")
		joblib.dump(user_sim,      f"{MODELS_DIR}/user_sim.pkl")
		joblib.dump(rating_avg,    f"{MODELS_DIR}/rating_avg.pkl")
	\end{lstlisting}
	
	\paragraph{Clasificación de Sentimientos.}
	Para capturar el tono de las opiniones de los usuarios más allá de la
	calificación numérica, se entrena un modelo de aprendizaje profundo basado
	en embeddings de palabras. Los comentarios se tokenizaron y convirtieron en
	secuencias de enteros de longitud fija (100 tokens). Las etiquetas binarias
	se derivan de las calificaciones: se considera positiva toda calificación
	estrictamente mayor a 3. El modelo consiste en una capa de embeddings de
	128 dimensiones, una capa de aplanamiento y una neurona de salida con
	activación sigmoide:
	
	\begin{lstlisting}[language=Python, caption=Tokenización y entrenamiento del modelo de sentimientos en train.py]
		# Tokenizacion
		tokenizer = Tokenizer()
		tokenizer.fit_on_texts(comments['comment'])
		sequences  = tokenizer.texts_to_sequences(comments['comment'])
		word_index = tokenizer.word_index
		data       = pad_sequences(sequences, maxlen=100)
		
		# Etiquetas binarias
		labels = np.array([
		1 if r > 3 else 0
		for r in ratings['rating'].head(len(data))
		])
		
		# Arquitectura del modelo
		model = Sequential([
		Embedding(input_dim=len(word_index) + 1,
		output_dim=128,
		input_length=100),
		Flatten(),
		Dense(1, activation='sigmoid'),
		])
		model.compile(optimizer='adam',
		loss='binary_crossentropy',
		metrics=['accuracy'])
		
		# Entrenamiento
		model.fit(data, labels,
		epochs=10,
		batch_size=32,
		validation_split=0.2,
		verbose=1)
		
		# Serializacion
		model.save(f"{MODELS_DIR}/keras_model.keras")
		with open(f"{MODELS_DIR}/tokenizer.pkl", "wb") as f:
		pickle.dump(tokenizer, f)
	\end{lstlisting}

	\subsubsection{Paso 5: Integración de Geolocalización}
	
	La geolocalización constituye el factor contextual principal del sistema.
	En \texttt{ALGORITMO\_PROPUESTO.py} se implementa la fórmula de Haversine
	para calcular la distancia en kilómetros entre el usuario y cada producto,
	considerando la curvatura de la Tierra. La función \texttt{add\_distance}
	acepta coordenadas explícitas cuando éstas provienen del dispositivo móvil,
	o bien las recupera del almacén de datos si no se proporcionan:
	
	\begin{lstlisting}[language=Python, caption=Cálculo de distancias geográficas en ALGORITMO\_PROPUESTO.py]
		def haversine(lat1, lon1, lat2, lon2):
		R    = 6371
		dlat = np.radians(lat2 - lat1)
		dlon = np.radians(lon2 - lon1)
		a = (np.sin(dlat / 2) ** 2
		+ np.cos(np.radians(lat1))
		* np.cos(np.radians(lat2))
		* np.sin(dlon / 2) ** 2)
		c = 2 * np.arctan2(np.sqrt(a), np.sqrt(1-a))
		return R * c
		
		
		def add_distance(user_id, lat_usuario=None, lon_usuario=None):
		if lat_usuario is not None and lon_usuario is not None:
		u_lat, u_lon = lat_usuario, lon_usuario
		else:
		user_loc = locations[locations['user_id'] == user_id]
		u_lat    = user_loc['latitude'].values[0]
		u_lon    = user_loc['longitude'].values[0]
		
		distances = [
		haversine(u_lat, u_lon, product['latitude'], product['longitude'])
		for _, product in products.iterrows()
		]
		
		products['distance'] = distances
		
		return u_lat, u_lon
	\end{lstlisting}

	\subsubsection{Paso 6: Generación de Recomendaciones}
	
	La función \texttt{recommend} en \texttt{ALGORITMO\_PROPUESTO.py} orquesta
	todos los componentes anteriores. Calcula los tres scores parciales
	(\textit{geo\_score}, \textit{rating\_score} y \textit{content\_score}),
	los normaliza al rango $[0,1]$, los combina con ponderaciones fijas y
	ordena los productos de mayor a menor score final. A cada resultado se le
	asigna una etiqueta explicativa mediante \texttt{generate\_reason}, que
	evalúa umbrales individuales por score para determinar las razones de
	recomendación presentadas al usuario:
	
	\begin{lstlisting}[language=Python, caption=Función de recomendación híbrida en ALGORITMO\_PROPUESTO.py]
		def recommend(user_id, lat_manual=None, lon_manual=None, top_n=5):
		u_lat, u_lon = add_distance(user_id, lat_manual, lon_manual)
		rec = products.copy()
		
		# Rating promedio
		rec = rec.merge(rating_avg.drop_duplicates('item_id'),
		on="item_id", how="left")
		rec["avg_rating"] = rec["avg_rating"].fillna(0)
		
		# Scores parciales
		rec["geo_score"]    = normalize_distance(rec["distance"])
		rec["rating_score"] = normalize_rating(rec["avg_rating"])
		
		# TF-IDF score
		rec["tfidf_score"] = [
		similarity_df[i].mean()
		if i in similarity_df.columns else 0
		for i in rec["item_id"]
		]
		
		# LDA score y sentiment score
		lda_score, sentiment_score = compute_text_scores()
		rec["lda_score"]       = rec.index.map(
		lambda i: lda_score[i] if i < len(lda_score) else 0)
		rec["sentiment_score"] = rec["item_id"].map(
		sentiment_score).fillna(0)
		
		# Normalizacion
		rec["tfidf_score"]     = normalize_rating(rec["tfidf_score"])
		rec["lda_score"]       = normalize_rating(rec["lda_score"])
		rec["sentiment_score"] = normalize_rating(rec["sentiment_score"])
		
		# Content score combinado
		rec["content_score"] = (0.5 * rec["tfidf_score"]
		+ 0.2 * rec["lda_score"]
		+ 0.3 * rec["sentiment_score"])
		
		# Score final ponderado
		rec["final_score"] = (0.40 * rec["geo_score"]
		+ 0.25 * rec["rating_score"]
		+ 0.35 * rec["content_score"])
		
		rec["reason"]   = rec.apply(generate_reason, axis=1)
		rec             = rec.sort_values("final_score", ascending=False)
		rec["distance"] = rec["distance"].apply(lambda x: f"{x:.2f} km")
		
		return rec[["name", "distance", "avg_rating", "reason"]].head(top_n)
		
		
		def generate_reason(row):
		reasons = []
		try:
		if float(row["geo_score"])     > 0.7:
		reasons.append("Cerca de tu ubicacion")
		if float(row["rating_score"])  > 0.6:
		reasons.append("Muy bien calificado por usuarios")
		if float(row["content_score"]) > 0.5:
		reasons.append("Similar a productos populares")
		except Exception as e:
		print(f"Error en generate_reason: {e}")
		return ", ".join(reasons) if reasons else "Recomendado por el sistema"
	\end{lstlisting}

	ordenados por score. Las coordenadas también son opcionales; de no
	suministrarse, el sistema utiliza valores fijos correspondientes al centro
	de la Ciudad de México como fallback:
	
	\begin{lstlisting}[language=Python, caption=Endpoint de recomendación en main.py]
		app = FastAPI()

		except Exception as e:
